# Portability Classification Ledger

**Bead:** `opencodehx-043`

This is the first source-level portability ledger for OpenCodeHX. It classifies the current Haxe source by the dominant retargeting seam and names the blockers that would matter for future `haxe.rust` or `haxe.go` experiments.

The file-level inventory is generated by:

```sh
npm run inventory:portability
```

Output lives in [portability-classification-ledger.csv](../reference/portability-classification-ledger.csv). The CSV is deliberately mechanical: it uses path ownership and import/use patterns to produce reviewable classifications. Treat it as a planning ledger, not as proof that any non-TypeScript target already compiles.

## Current Shape

The strongest portable core today is DTO/model and deterministic transformation code: message/session identifiers, session compaction/retry rules, permission rules, event bus, sync store semantics, many provider transforms, CLI surface cataloging, TUI route/keybind/theme helpers, and small utilities.

The largest retargeting cost is not Haxe syntax. It is host ownership. The project intentionally preserves OpenCode's Node-first behavior, so many product modules currently cross Node APIs directly or through narrow host facades: project/worktree/git, file tools, package-manager runtime, SQLite stores, PTY, server startup, config discovery, skills, and workspace sync.

The TS-only surface is visible but bounded. Most `genes.ts` and `@:ts.*` uses are either SDK type bridges, TS declaration quality helpers, or JS runtime identity helpers. They are acceptable for the TypeScript target, but a Rust/Go experiment needs target-neutral semantic types before those files can move.

## Rust/Go Blockers

1. **Node host adapter coverage.** Rust/Go would need equivalents for `NodePath`, `NodeProcess`, `Fs`, `ChildProcess`, `Crypto`, `Os`, `Url`, `node-pty`, git execution, and package-manager process seams. The app-facing APIs are already narrow in several places, but some product modules still import Node facades directly.
2. **Async and stream model.** Many generated TypeScript paths use `js.lib.Promise`, web `ReadableStream`, `AbortController`, WebSocket events, and AI SDK streams. A future target needs a small Haxe-owned async/stream facade before product logic can be retargeted cleanly.
3. **npm SDK dependency surface.** Provider loading, AI SDK language models, Hono, OpenTUI/Solid, tree-sitter, JSONC/TOML parsers, AWS credential providers, and `better-sqlite3` are npm-owned. Rust/Go should either replace these behind existing facades or keep them in a sidecar/TS adapter.
4. **Storage and database adapters.** `SessionStore` is portable, but concrete SQLite persistence currently depends on `better-sqlite3`. Rust/Go need native SQLite adapters with the same session/sync store contracts and fixture evidence.
5. **Resource packaging.** Current resource imports are copied into TypeScript package paths and include text, JSON, worker files, WAV fixtures, and tree-sitter WASM. Rust/Go targets need deterministic embedded or sidecar resource packaging before file/tool/TUI parity can be claimed.
6. **Generated-TS helper types.** `genes.ts.Unknown`, `Undefinable`, import helpers, erased TS-only declarations, and raw `@:ts.type(...)` bridges need target-polymorphic equivalents or target guards.
7. **Raw JS boundaries.** Remaining `js.Syntax.code`, `untyped`, JS identity-key behavior, and JS `Error`/`Uint8Array` assumptions must either be retired into typed facades or marked as target-specific adapter code.
8. **TUI renderer.** The TUI model helpers are mostly portable, but `TuiScaffold` is OpenTUI/Solid TSX. A Rust/Go target needs a renderer decision before live TUI parity can move.

## Prioritized Experiments

1. **Portable-core compile probe.** Create a narrow Haxe target set for `portable` files only, excluding `smoke`, `externs`, `host/node`, provider SDK loaders, and TSX. First target can be a non-production compile probe; success means the domain core is syntactically retargetable, not behavior-complete.
2. **Host contract extraction.** Promote direct Node usage in project/file/tool/skill/config/runtime modules into explicit Haxe interfaces for filesystem, process, path, git, HTTP, timers, crypto, and resource reads. Keep Node implementations as the production default.
3. **Async facade audit.** Decide whether `opencodehx.fx.Task` should become the target-neutral async contract or stay an Effect compatibility shim. Then move app-facing Promise/stream code behind that contract where it is not inherently SDK-owned.
4. **SQLite adapter spike.** Implement an injected store persistence interface for session and sync SQLite paths, then keep `better-sqlite3` as one adapter. This is the cleanest first real host swap.
5. **Resource packaging spike.** Generate a manifest that can emit both TypeScript package paths and an embedded/sidecar target profile. Prove it with text JSON, binary WAV, and WASM resource reads.
6. **Provider sidecar boundary.** Treat AI SDK providers as a TS sidecar candidate and define the narrow request/stream events that Rust/Go would consume. Avoid porting every npm provider package.
7. **TUI renderer decision.** Separate route/theme/transcript state from OpenTUI/Solid rendering so a future target can reuse the state model without TSX.
8. **Retargeted smoke intent.** Use the Haxe-authored smokes as the source of future target tests, but regenerate host-specific runners instead of trying to run Node smoke code unchanged.

## Maintenance

Refresh the CSV when files move, new host seams appear, or a module is made portable:

```sh
npm run inventory:portability
```

When closing future portability work, update this doc if a class definition changes, a blocker disappears, or a priority changes. Do not use this ledger to justify weakening TypeScript output quality; compiler and TS surface quality remain `genes-ts` product gates.
